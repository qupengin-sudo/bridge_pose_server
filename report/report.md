# 第 1 章 概述

## 1.1 專題動機

橋式（Bridge Exercise）是一種常見的核心肌群訓練動作，透過仰臥後抬起臀部，鍛鍊臀大肌、腿後肌群及下背部肌肉，對於改善姿勢、減緩下背痛具有顯著效果。然而在居家自主訓練的情境下，使用者往往缺乏專業教練的即時指導，容易因動作不標準而降低訓練效果，甚至造成運動傷害。

市面上雖有不少健身輔助應用程式，但多數需要手動計數或佩戴額外穿戴裝置。手動計數容易因分心而遺漏，穿戴裝置則增加了使用門檻與成本。若能透過攝影機即時辨識使用者的動作姿勢，並自動判斷橋式動作是否完成，不僅能減少人為計數的誤差，也能讓使用者專注於動作本身，提升訓練品質。

因此，本專題旨在開發一套「橋式健身輔助系統」，以攝影機搭配姿態辨識技術，即時偵測使用者的橋式動作，自動計算動作次數與訓練時間，並將成績儲存於資料庫中，供使用者日後查看與追蹤訓練進度。

## 1.2 專題目標

本專題的核心目標是實現即時姿態辨識與訓練紀錄管理，使使用者能在無人指導的情況下完成有效的橋式訓練。具體功能如下：

- 即時偵測橋式動作：透過攝影機擷取畫面，利用姿態辨識模型標記人體關鍵點，計算關節角度以判斷橋式動作階段。

- 自動計數與計時：以有限狀態機管理動作階段轉換，當使用者完成一次完整的「抬臀→放下」循環即自動累計一次，同時支援使用者自訂訓練時間。

- 帳號系統與成績存檔：提供註冊、登入功能，訓練結束後自動將成績存入資料庫。

- 數據視覺化與歷史查詢：以圖表呈現每日訓練統計，並提供分頁瀏覽與進階搜尋功能，方便使用者回顧訓練歷程。

## 1.3 情境描述

小明是一位久坐辦公室的上班族，醫師建議他每天做橋式運動來改善下背痛。小明決定在家中自主訓練，但不確定自己的動作是否標準，也經常忘記計數。

小明的操作流程：

- 小明打開電腦，在瀏覽器中輸入系統網址，進入橋式健身輔助系統的首頁。

- 他點擊「登入」按鈕，輸入帳號與密碼後成功登入。若是第一次使用，他可以先點擊「註冊」建立帳號。

- 登入後，小明點擊「開始運動」按鈕，系統彈出計時器選擇視窗，他選擇訓練 3 分鐘。

- 攝影機畫面隨即顯示在螢幕上，畫面左上角有一個資訊面板，顯示目前的動作階段、完成次數與倒數計時。計時尚未開始，提示「做出橋式動作即開始計時」。

- 小明躺在瑜珈墊上，擺好姿勢後抬起臀部，系統偵測到橋式動作，畫面上的骨架線條變為綠色，狀態顯示「STAGE1」，計時同時啟動。

- 小明緩緩放下臀部，系統偵測角度變化，狀態經過「TRANSITION」後進入「STAGE2」，計數增加 1 次。

- 小明持續重複動作，每完成一次「抬起→放下」的循環，計數自動加 1。

- 3 分鐘結束後，攝影機自動關閉，螢幕彈出訓練結果對話框，顯示他共完成了 15 次橋式。

- 系統自動將這筆紀錄存入資料庫。小明可以在「查看所有紀錄」頁面看到每日的訓練圖表與歷史紀錄列表。

訪客使用：

- 即使不登入，小明也可以直接點擊「開始運動」進行訓練。運動結束後，系統會提示他登入或註冊以保存紀錄。

- 若小明此時登入，系統會自動將剛才的成績補存至他的帳號下。

# 第 2 章 相關知識及技術

本章介紹建構本系統所使用的技術與工具，包括姿態辨識框架、後端框架、前端技術、資料庫，以及桌面介面工具。這些技術的介紹將為後續章節的系統架構和實作說明奠定基礎。

## 2.1 MediaPipe Pose

MediaPipe 是 Google 開發的一個開源跨平台框架，提供多種預訓練的機器學習解決方案，包括人臉偵測、手部追蹤、物件偵測等。其中 MediaPipe Pose 模組能夠即時標記人體 33 個 Keypoints（關鍵點），這些關鍵點對應人體結構上的特定位置，例如肩膀、髖部、膝蓋、腳踝等關節。

> 【此處需要圖片：MediaPipe 人體 33 個 Keypoints 對應圖】

本專題使用 MediaPipe 的 Pose Landmarker Lite 模型（pose_landmarker_lite.task），以 VIDEO 模式逐幀處理攝影機影像。模型會回傳每個關鍵點的座標（x, y, z）與可見度（visibility），系統據此取出所需的關節位置進行角度計算。

本系統主要使用以下四個關鍵點（以左側為例）：

| Keypoints 編號 | 名稱 | 用途 |
|:---:|:---:|:---:|
| 11 | left_shoulder | 肩膀位置，用於計算膝-髖-肩角度 |
| 23 | left_hip | 髖部位置，作為角度計算的頂點 |
| 25 | left_knee | 膝蓋位置，用於判斷腿部彎曲程度 |
| 31 | left_foot_index | 腳踝位置，用於輔助角度計算 |

系統會優先偵測左側身體，若左側關鍵點可見度不足（低於 0.5），則自動切換至右側（編號 12、24、26、32）。

## 2.2 OpenCV

OpenCV（Open Source Computer Vision Library）是一個開源的電腦視覺函式庫，提供豐富的影像處理與分析功能。本系統使用 OpenCV 實現以下功能：

- 攝影機影像擷取：透過 `cv2.VideoCapture` 開啟攝影機並逐幀讀取影像。

- 影像格式轉換：將 BGR 格式轉為 RGB 格式以供 MediaPipe 處理。

- 骨架繪製：在影像上繪製關節圓點、骨架連線、角度弧線與角度數值標註。

- 視窗管理：建立可調整大小的顯示視窗，並將其置中於主螢幕。

## 2.3 FastAPI

FastAPI 是一個現代化的 Python Web 框架，基於 Starlette 和 Pydantic 建構，具有高效能與自動產生 API 文件的特性。本系統以 FastAPI 作為後端伺服器，搭配 Uvicorn 作為 ASGI 伺服器運行。FastAPI 負責處理前端發送的 HTTP 請求，包括使用者註冊、登入、啟動偵測程式、儲存運動紀錄及查詢統計資料等功能。

## 2.4 SQLite

SQLite 是一個輕量級的嵌入式關聯式資料庫，不需要獨立的伺服器進程，資料庫以單一檔案（users.db）儲存，適合小型應用程式使用。本系統使用 SQLite 儲存使用者帳號資訊與運動紀錄，透過 Python 內建的 `sqlite3` 模組進行操作。

## 2.5 Chart.js

Chart.js 是一個開源的 JavaScript 圖表函式庫，能夠在網頁上繪製互動式圖表。本系統使用 Chart.js 繪製「每日運動紀錄堆疊長條圖」，將使用者每日的訓練次數以不同顏色堆疊呈現，滑鼠移至長條上可顯示該次訓練的詳細資訊（次數與時長）。

## 2.6 Tkinter

Tkinter 是 Python 的標準 GUI（圖形使用者介面）工具組，無須額外安裝即可使用。本系統使用 Tkinter 建立三種桌面對話框：

- 訓練時間選擇器：使用者在開始訓練前選擇分鐘與秒數。

- 訓練結果對話框：訓練結束後顯示完成次數與訓練時間。

- 錯誤提示對話框：當使用者未完成任何動作即關閉視窗時顯示提示訊息。

所有 Tkinter 視窗均使用 `xrandr` 偵測主螢幕位置，確保在多螢幕環境下正確置中於主螢幕。

# 第 3 章 系統簡介

本章說明系統的整體架構與運作流程，呈現系統如何實現橋式動作偵測、自動計數與紀錄管理功能。除了介紹系統的各個組成部分及其功能之外，也會說明系統各模組之間如何互動以達成訓練輔助之目的。

## 3.1 系統架構

本系統採用前後端整合架構，包含 Web 伺服器、姿態偵測模組、資料庫及前端頁面四個主要部分。

> 【此處需要圖片：系統架構圖，標示 Web 伺服器、姿態偵測模組、資料庫、前端頁面之間的關係】

### 3.1.1 Web 伺服器（FastAPI）

Web 伺服器為系統的核心樞紐，負責接收前端請求並協調各模組運作。主要功能包括：

- 頁面服務：使用 Jinja2 模板引擎渲染首頁 HTML 頁面。

- 帳號管理：處理使用者的註冊、登入、登出請求，以 Cookie 維持登入狀態（有效期 30 天）。

- 偵測啟動：接收前端的「開始運動」請求後，呼叫姿態偵測模組開啟攝影機進行即時偵測。

- 紀錄管理：訓練結束後將成績寫入資料庫，並提供統計查詢與搜尋 API。

### 3.1.2 姿態偵測模組（bridge_detector.py）

姿態偵測模組是系統的核心運算單元，負責開啟攝影機、載入 MediaPipe 模型、即時偵測橋式動作，並透過有限狀態機管理動作階段。當訓練時間結束或使用者手動關閉視窗時，模組回傳訓練時長與完成次數給 Web 伺服器。

### 3.1.3 資料庫（SQLite）

資料庫以單一檔案 `users.db` 儲存所有資料，包含兩張資料表：

- `users` 表：儲存使用者帳號與密碼。

- `records` 表：儲存每次訓練的使用者名稱、正確次數、錯誤次數、訓練時長與時間戳。

伺服器啟動時會自動呼叫 `init_all_tables()` 建立資料表（若尚未存在），確保系統隨時可用。

### 3.1.4 前端頁面（index.html）

前端為單頁式網頁應用，包含以下區塊：

- 主頁：顯示「開始運動」按鈕，右上角顯示登入狀態與操作按鈕。

- 帳號管理頁：提供註冊與登入表單。

- 紀錄頁：上方為每日運動統計堆疊長條圖，下方為紀錄列表（含搜尋與分頁功能）。

## 3.2 系統流程

### 3.2.1 使用者操作流程

使用者的完整操作流程如下：

1. 開啟瀏覽器，進入系統首頁（http://127.0.0.1:8000）。

2. （選擇性）點擊「登入/註冊」進行帳號管理。

3. 點擊「開始運動」按鈕，前端發送 POST 請求至 `/open_app`。

4. 系統彈出訓練時間選擇器（Tkinter 對話框），使用者選擇訓練時長後點擊「開始訓練」。

5. 攝影機畫面開啟，等待使用者做出第一個橋式動作後開始倒數計時。

6. 使用者重複「抬臀→放下」的動作，系統自動計數。

7. 倒數結束後攝影機關閉，彈出訓練結果對話框。

8. 若使用者已登入，成績自動存入資料庫；若為訪客，紀錄暫存於前端，待登入後自動補存。

9. 使用者可在紀錄頁查看圖表與歷史紀錄。

> 【此處需要圖片：使用者操作流程圖】

### 3.2.2 橋式動作辨識

系統透過計算膝-髖-肩（Knee-Hip-Shoulder）角度來判斷橋式動作的階段。以髖部（Hip）為頂點，計算膝蓋（Knee）與肩膀（Shoulder）之間形成的角度。

> 【此處需要圖片：橋式動作角度示意圖，標示 Shoulder、Hip、Knee 三點與角度】

角度計算公式如下：

給定三個點 A（Knee）、B（Hip）、C（Shoulder），以 B 為頂點的角度為：

$$\theta = \arccos\left(\frac{\vec{BA} \cdot \vec{BC}}{|\vec{BA}| \times |\vec{BC}|}\right)$$

根據角度數值，動作分為三個階段：

| 階段 | 角度範圍 | 顏色 | 說明 |
|:---:|:---:|:---:|:---|
| STAGE1（橋式完成） | 155°–180° | 綠色 | 臀部抬起，身體呈接近一直線 |
| TRANSITION（過渡） | 131°–154° | 紅色 | 臀部正在抬起或放下的中間狀態 |
| STAGE2（休息） | ≤ 130° | 橘色 | 臀部放下，回到休息位置 |

上述角度範圍可透過 `config.json` 設定檔調整，以適應不同使用者的身體比例。

### 3.2.3 狀態機轉換

系統使用有限狀態機（Finite State Machine, FSM）管理動作階段的轉換，確保計數邏輯的正確性。狀態機包含四個狀態：

> 【此處需要圖片：狀態機轉換圖，標示 INIT、STAGE1、TRANSITION、STAGE2 四個狀態及轉換條件】

- **INIT（初始狀態）**：系統啟動後的等待狀態。當偵測到使用者進入 STAGE1（角度 155°–180°）時，狀態轉換至 STAGE1 並啟動計時器。

- **STAGE1（橋式完成）**：使用者臀部完全抬起。若角度下降至 TRANSITION 範圍，轉換至 TRANSITION；若角度直接下降至 STAGE2 範圍，轉換至 STAGE2 並計數加 1。

- **TRANSITION（過渡階段）**：使用者臀部在抬起與放下之間。若角度回升至 STAGE1 範圍，轉換回 STAGE1；若角度繼續下降至 STAGE2 範圍，轉換至 STAGE2 並計數加 1。

- **STAGE2（休息狀態）**：使用者臀部完全放下，一次完整的橋式動作計數完成。若角度回升至 TRANSITION 或 STAGE1 範圍，轉換至對應狀態，開始下一次動作。

計數規則：每當狀態從 STAGE1 或 TRANSITION 轉換至 STAGE2 時，代表使用者完成了一次「抬臀→放下」的完整循環，此時 bridge_count 加 1。

### 3.2.4 訪客機制

系統允許未登入的訪客直接使用訓練功能。訪客完成訓練後，成績會暫存於前端的 `pendingRecord` 變數中。若訪客隨後登入或註冊，系統會自動透過 `/save_record` API 將暫存的紀錄補存至該帳號下。此設計降低了使用門檻，讓使用者可以先體驗再決定是否建立帳號。

# 第 4 章 系統實作

本章將說明本系統的建置與實作過程，包括各模組的程式碼結構與功能概述。系統由後端伺服器、姿態偵測核心、資料庫初始化、前端頁面四個部分組成，以下分別介紹各部分的實作細節。

## 4.1 專案檔案結構

```
bridge_pose_server/
├── app.py                    # FastAPI 主程式，定義所有 API 端點
├── bridge_detector.py        # 姿態偵測核心模組（攝影機、MediaPipe、FSM）
├── init_record.py            # 資料庫初始化（建立 users 與 records 資料表）
├── config.json               # 設定檔（攝影機編號、角度閾值）
├── pose_landmarker_lite.task  # MediaPipe Pose Landmarker 模型檔案
├── templates/
│   └── index.html            # 前端單頁式網頁
└── users.db                  # SQLite 資料庫檔案（首次啟動自動建立）
```

## 4.2 設定檔（config.json）

系統的可調整參數集中於 `config.json`，在 `bridge_detector.py` 啟動時載入。各參數說明如下：

```json
{
    "DEVICE_ID": 0,
    "STAGE1_MIN": 155,
    "STAGE1_MAX": 180,
    "STAGE2_MAX": 130
}
```

| 參數 | 型別 | 預設值 | 說明 |
|:---:|:---:|:---:|:---|
| DEVICE_ID | 整數 | 0 | 攝影機裝置編號，可依連接的攝影機調整 |
| STAGE1_MIN | 整數 | 155 | STAGE1（橋式完成）的最小膝-髖-肩角度 |
| STAGE1_MAX | 整數 | 180 | STAGE1（橋式完成）的最大膝-髖-肩角度 |
| STAGE2_MAX | 整數 | 130 | STAGE2（休息）的最大角度，低於此值為休息階段 |

TRANSITION 的角度範圍由 STAGE2_MAX 與 STAGE1_MIN 之間的區間自動推算，即 131°至 154°。

## 4.3 資料庫實作（init_record.py）

資料庫初始化模組負責建立兩張資料表。伺服器啟動時會自動呼叫 `init_all_tables()`，使用 `CREATE TABLE IF NOT EXISTS` 確保資料表存在但不覆蓋既有資料。

### 4.3.1 users 資料表

儲存使用者帳號資訊。

| 欄位 | 型別 | 約束 | 說明 |
|:---:|:---:|:---:|:---|
| id | INTEGER | PRIMARY KEY AUTOINCREMENT | 自動遞增的唯一識別碼 |
| username | TEXT | UNIQUE NOT NULL | 使用者帳號，不可重複 |
| password | TEXT | NOT NULL | 使用者密碼 |

### 4.3.2 records 資料表

儲存每次訓練的成績資料。

| 欄位 | 型別 | 約束 | 說明 |
|:---:|:---:|:---:|:---|
| id | INTEGER | PRIMARY KEY AUTOINCREMENT | 自動遞增的唯一識別碼 |
| username | TEXT | NOT NULL | 對應的使用者帳號 |
| correct_count | INTEGER | — | 該次訓練的正確橋式次數 |
| error_count | INTEGER | — | 錯誤次數（目前預留，固定為 0） |
| duration_seconds | INTEGER | — | 訓練時長（單位：秒） |
| timestamp | DATETIME | DEFAULT CURRENT_TIMESTAMP | 紀錄建立時間（UTC） |

## 4.4 後端伺服器實作（app.py）

app.py 為系統的後端主程式，使用 FastAPI 框架定義所有 API 端點。以下概述各端點的功能。

### 4.4.1 首頁與帳號管理 API

| 方法 | 路徑 | 功能說明 |
|:---:|:---:|:---|
| GET | `/` | 使用 Jinja2 渲染並回傳首頁 index.html |

## 4.5 姿態偵測核心實作（bridge_detector.py）

bridge_detector.py 是系統的運算核心，負責攝影機控制、姿態辨識、狀態管理與使用者介面。以下依功能分段說明。

### 4.5.1 設定檔載入與主螢幕偵測

模組啟動時會從 `config.json` 載入攝影機編號與角度閾值。同時提供 `get_primary_monitor()` 函式，透過執行 `xrandr` 指令解析主螢幕的位置與解析度，供後續視窗置中使用。此設計確保在多螢幕環境下，所有對話框與攝影機畫面均正確顯示於主螢幕中央。

### 4.5.2 角度計算函式

`calculate_angle(a, b, c)` 函式接收三個二維座標點，以 b 為頂點計算 a-b-c 形成的角度（單位：度）。內部使用向量內積公式，並以 `np.clip` 限制 cos 值範圍以避免浮點數誤差導致的計算錯誤。

### 4.5.3 橋式偵測函式

`detect_bridge(shoulder, hip, knee, ankle)` 函式接收四個關節的像素座標，計算三個角度：

- **a_khs**（Knee-Hip-Shoulder）：主要的橋式判斷角度，以髖部為頂點。此角度在橋式完成時接近 180°，休息時角度較小。

- **a_fkh**（Ankle-Knee-Hip）：腿部彎曲的參考角度。

- **a_kfs**（Knee-Ankle-Shoulder）：腳部相對肩膀的參考角度。

函式根據 a_khs 的數值判斷是否處於 STAGE1（橋式完成）或 TRANSITION（過渡階段），並回傳判斷結果與各角度值。

### 4.5.4 訓練時間選擇器（ask_duration）

`ask_duration()` 函式使用 Tkinter 建立一個深色主題的對話框，包含分鐘與秒數的 Spinbox 選擇器及「開始訓練」按鈕。分鐘範圍為 0–59 分，秒數以 10 秒為增量（0–50 秒）。

> 【此處需要圖片：訓練時間選擇器介面截圖】

使用者選擇時間後點擊「開始訓練」，函式回傳總秒數與螢幕資訊。若使用者直接關閉視窗未選擇時間，程式會直接結束。

### 4.5.5 主迴圈（open_bridge_app）

`open_bridge_app()` 為偵測流程的入口函式，由 Web 伺服器呼叫。其執行流程如下：

1. **初始化 MediaPipe 模型**：載入 Pose Landmarker Lite 模型，設定為 VIDEO 模式。

2. **呼叫時間選擇器**：取得使用者設定的訓練時長。

3. **開啟攝影機**：以 `cv2.VideoCapture` 開啟指定裝置。

4. **設定視窗大小**：先讀取一幀取得攝影機解析度，以主螢幕高度的 70% 為基準按比例計算視窗大小，並置中顯示。

5. **進入主迴圈**：

   - 逐幀讀取攝影機影像。

   - 將影像轉為 RGB 格式後送入 MediaPipe 模型。

   - 從模型結果中提取肩膀、髖部、膝蓋、腳踝四個關鍵點的像素座標。

   - 呼叫 `detect_bridge()` 計算角度並判斷動作階段。

   - 執行 FSM 狀態轉換邏輯，更新計數與計時。

   - 在影像上繪製骨架（關節點、連線）、輔助線（腳踝至肩膀的虛線）與角度弧線。

   - 繪製左上角的資訊面板（Mini Sub-window），顯示狀態、計數、角度、倒數計時與骨架縮圖。

   - 顯示影像並檢查退出條件。

> 【此處需要圖片：攝影機畫面截圖，標示骨架繪製、角度標註與左上角資訊面板】

6. **結束處理**：

   - 倒數結束：釋放攝影機，彈出訓練結果對話框，回傳總訓練時間與完成次數。

   - 使用者手動關閉（按 q 鍵或點擊視窗 X 按鈕）：計算實際經過時間，若有完成次數則彈出結果對話框，否則彈出錯誤提示對話框。

### 4.5.6 資訊面板（Mini Sub-window）

資訊面板位於攝影機畫面的左上角，尺寸為 230×290 像素，包含三個區塊：

- **頂部**：顯示目前階段名稱（以對應顏色標示）、累計次數（青色粗體）與 KHS 角度值。

- **中部**：顯示骨架縮圖，將攝影機畫面中的人體姿勢映射至固定大小的示意圖。腳踝與肩膀分別固定於面板的左下角與右下角，髖部與膝蓋根據實際姿勢按比例計算位置。縮圖同時繪製輔助虛線與角度弧線。

- **底部**：顯示倒數計時（格式 MM:SS），最後 10 秒以紅色閃爍提示。若計時尚未開始，顯示「do pose to start」。

> 【此處需要圖片：資訊面板放大截圖，標示三個區塊】

### 4.5.7 訓練結果對話框

訓練結束後彈出的結果對話框為深色主題的 Tkinter 視窗，顯示以下資訊：

- 標題「訓練完成！」

- 完成次數（大字綠色數字）

- 單位標示「次橋式 / bridge reps」

- 訓練時間（分鐘）

- 「關閉」按鈕

> 【此處需要圖片：訓練結果對話框截圖】

### 4.5.8 錯誤提示對話框

當使用者在未完成任何橋式動作的情況下關閉攝影機視窗時，系統會彈出錯誤提示對話框，顯示「系統發生錯誤或您已手動關閉視窗。」訊息，提醒使用者此次訓練未產生有效紀錄。

> 【此處需要圖片：錯誤提示對話框截圖】

## 4.6 前端實作（index.html）

前端為單頁式 HTML 網頁，使用原生 JavaScript 與 Fetch API 與後端通訊，搭配 Chart.js 繪製圖表。以下說明各功能區塊的實作。

### 4.6.1 頁面結構

頁面分為三個主要區塊，任一時間僅顯示其中一個：

- **主頁（main_content）**：包含標題「橋式健身輔助系統」與「開始運動」按鈕。右上角顯示登入狀態（訪客/使用者名稱）、登入/登出按鈕與「查看所有紀錄」按鈕。

- **帳號管理頁（auth_page）**：提供「我要註冊」與「我要登入」按鈕，點擊後動態產生對應的輸入表單（帳號、密碼欄位與送出按鈕）。

- **紀錄頁（records_page）**：上半部為每日運動統計圖表，下半部為歷史紀錄列表。

> 【此處需要圖片：系統首頁截圖】

### 4.6.2 每日運動統計圖表

圖表使用 Chart.js 繪製堆疊長條圖。使用者可透過「顯示天數」輸入框調整回顧天數（1–50 天），輸入後系統以 400 毫秒的防抖機制避免頻繁請求。

圖表的 X 軸為日期（格式 MM/DD），Y 軸為正確次數。同一天的多次訓練以不同顏色堆疊呈現，滑鼠移至長條上會顯示「第 N 次：X 次 (M分S秒)」的詳細資訊。

> 【此處需要圖片：每日運動統計堆疊長條圖截圖】

### 4.6.3 歷史紀錄列表

紀錄列表以表格形式呈現，欄位包括序號、日期時間、正確次數、錯誤次數與運動時長。每頁顯示 20 筆紀錄，下方提供分頁導航按鈕。

列表上方設有搜尋列，支援以下篩選條件：

- **日期範圍**：選擇起始與結束日期。

- **運動時長**：選擇比較方式（等於 / 小於等於 / 大於等於）並輸入分鐘數。

點擊「搜尋」後系統向 `/search_records` API 發送請求，取得篩選結果。點擊「重置」可清除篩選條件回到全部紀錄。

> 【此處需要圖片：歷史紀錄列表與搜尋功能截圖】

### 4.6.4 訪客暫存機制

前端使用 `pendingRecord` 變數暫存訪客的訓練成績。當訪客完成訓練後，`pendingRecord` 會記錄 `{ duration, count }`。若訪客隨後登入，`handleLogin()` 函式會檢查 `pendingRecord` 是否存在，若存在則自動呼叫 `/save_record` API 補存紀錄，並清除暫存。

## 4.7 系統操作流程

### 4.7.1 環境準備

在使用本系統前，需要完成以下準備工作：

1. 確認電腦已安裝 Python 3.x 環境。

2. 安裝必要的 Python 套件：

```bash
pip install fastapi uvicorn mediapipe opencv-python numpy jinja2
```

3. 確認攝影機已正確連接，並在 `config.json` 中設定正確的 `DEVICE_ID`（可使用 0、1、2 等數值嘗試）。

4. 確認專案資料夾中包含 `pose_landmarker_lite.task` 模型檔案。

### 4.7.2 系統啟動

1. 開啟終端機，切換至專案資料夾：

```bash
cd bridge_pose_server
```

2. 執行主程式啟動伺服器：

```bash
python app.py
```

3. 伺服器會在 `http://127.0.0.1:8000` 啟動，終端機顯示 Uvicorn 的啟動訊息。首次啟動時系統會自動建立 `users.db` 資料庫檔案。

4. 開啟瀏覽器，輸入 `http://127.0.0.1:8000`，即可看到系統首頁。

### 4.7.3 開始訓練

1. 在首頁點擊「開始運動」按鈕，按鈕會變為「視訊啟動中...」。

2. 畫面彈出訓練時間選擇器，使用 Spinbox 選擇分鐘與秒數後，點擊「開始訓練」。

3. 攝影機畫面開啟，左上角資訊面板顯示計時尚未開始。

4. 使用者面對攝影機做出橋式動作（仰臥抬臀），系統偵測到後開始計時。

5. 使用者重複動作，觀察資訊面板中的計數與狀態變化，確認動作是否被正確辨識。

6. 倒數結束後自動彈出結果對話框，或按 q 鍵 / 點擊 X 按鈕提前結束。

### 4.7.4 角度閾值調整

若發現系統對動作的判斷過於寬鬆或嚴格，可以調整 `config.json` 中的角度閾值：

- 若使用者臀部已充分抬起但系統未判定為 STAGE1，可適當降低 `STAGE1_MIN` 的值。

- 若使用者臀部尚未完全放下即被判定為 STAGE2，可適當降低 `STAGE2_MAX` 的值。

修改設定檔後需重新啟動伺服器才會生效。

# 第 5 章 結論與未來發展

## 5.1 結論

本系統實現了一套即時橋式動作辨識與訓練紀錄管理的健身輔助工具。透過 MediaPipe 姿態辨識模型擷取人體關鍵點，計算膝-髖-肩角度以判斷橋式動作階段，搭配有限狀態機管理「抬臀→放下」的完整循環，自動累計動作次數。系統提供了使用者帳號管理、自訂訓練時間、自動存檔、每日統計圖表與歷史紀錄查詢等功能，讓使用者能在無人指導的情況下完成有效的居家橋式訓練。

## 5.2 未來發展

本系統仍有許多可完善的方向，以下列出幾項改進構想。

### 5.2.1 動作品質評估

目前系統僅判斷使用者是否完成橋式動作（角度是否達標），但並未評估動作的品質，例如臀部是否歪斜、膝蓋是否內扣等。未來可以利用 MediaPipe 提供的更多關鍵點（如左右髖部的高度差），加入動作品質評分機制，在畫面上即時提示使用者調整姿勢。

### 5.2.2 支援更多運動類型

目前系統僅支援橋式動作的辨識。由於 MediaPipe 能夠標記全身 33 個關鍵點，理論上可以擴展至其他運動動作的辨識，例如深蹲、平板支撐、仰臥起坐等。可以設計一套通用的動作定義框架，讓使用者或開發者自行定義各動作的關節角度條件。

### 5.2.3 帳號安全性強化

目前系統的帳號密碼以明文儲存於資料庫中，且未實作任何加密機制，存在安全風險。未來應導入密碼雜湊（如 bcrypt）機制，確保密碼不以明文形式儲存。此外，可引入 JWT（JSON Web Token）取代 Cookie 進行身份驗證，提升安全性。

### 5.2.4 行動裝置支援

目前系統需要在電腦端使用攝影機進行偵測，使用場景受限。未來可考慮將前端改為響應式設計，支援手機瀏覽器操作，並透過手機攝影機進行姿態偵測，使使用者能在任何地點進行訓練。
